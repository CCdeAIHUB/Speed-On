# ADR: Search and logging contracts

## 背景

Speed-On 需要从前端接收搜索内容，搜索应用、文件、文件夹、浏览器地址以及浏览器地址名称等信息。同时，系统需要记录用户最终选择打开了哪个结果，并把历史选择用于下一次搜索排序。

搜索和日志都涉及敏感数据：用户输入、文件路径、浏览器 URL、浏览器页面标题、运行期错误、系统状态等。如果把用户操作日志和系统日志混在一起，后续会很难做隐私控制、导出、清理和调试。

## 决策

1. 新增独立 `search` 模块，负责搜索归一化、匹配、排序和去重。
2. 新增独立 `logging` 模块，负责用户操作日志和系统日志模型。
3. 新增 `SearchIndexRepository`，由 SQLite 实现提供搜索候选、别名和历史选择信号。
4. 新增 `UserOperationLogRepository`，记录前端搜索 query 和最终打开选择。
5. 新增 `SystemLogSink`，记录运行期系统日志和错误诊断。
6. SQLite schema 升级到 v2，增加搜索别名、用户搜索日志、用户选择日志、查询-资源选择聚合和系统日志表。
7. 拼音搜索通过 `resource_search_aliases` 里的 `pinyin_full` 和 `pinyin_initials` 表达，不在搜索服务里硬编码拼音字典。
8. 用户历史选择排序权重高于普通标题、路径、URL、浏览器标题和拼音别名匹配。
9. 搜索结果必须按 `resource_id` 去重，同一个内容不能因历史命中和文本命中同时出现两次。

## 原因

- 搜索服务保持平台无关、数据库无关，方便 TDD。
- 拼音生成质量后续可以替换，不影响搜索服务契约。
- 用户日志和系统日志分表，便于隐私策略、清理策略和调试策略分别演进。
- 查询-资源选择聚合表可以避免每次按键搜索都全表扫描用户选择日志。
- 去重规则可以防止同一资源因多种匹配来源重复展示。

## 替代方案

### 方案 A：直接引入拼音库并在搜索服务里转换

优点是短期能直接生成拼音，但会把分词/拼音字典选择绑定到核心搜索逻辑，后续切换库或优化词典会影响搜索服务。

### 方案 B：只做全文搜索，不记录用户选择

实现简单，但无法满足“用户上次相近输入打开过的内容优先”的需求。

### 方案 C：搜索别名 + 用户选择聚合

这是当前选择。搜索服务只处理已归一化候选，SQLite 或后续索引构建器负责维护 title、target、browser_title、pinyin_full、pinyin_initials 等别名。

## 影响范围

- 新增 `logging.rs`。
- 新增 `search.rs`。
- 扩展 `ports.rs`。
- 扩展公开导出 `lib.rs`。
- SQLite schema 从 v1 升级到 v2。
- 新增搜索服务测试和 schema 测试。

## 风险

- 当前没有真实 SQLite repository，搜索候选和用户日志聚合仍由后续实现。
- 当前没有真实拼音生成器，拼音别名需要后续索引阶段填充。
- 用户搜索日志和选择日志包含隐私数据，后续必须增加保留周期、清理策略和用户开关。
- 系统日志必须保持脱敏，不能直接复制原始 query、完整 URL、token、密码或私有文件内容。

## 未来演进

1. 实现 SQLite repository，持久化搜索别名、用户日志和系统日志。
2. 增加 alias builder，生成 title、target、browser_title、pinyin_full、pinyin_initials。
3. 设计用户隐私设置：是否记录搜索、是否记录浏览器历史、日志保留周期。
4. 接入前端 IPC/API：search(query, limit, kinds) 与 record_selection。
5. 增加系统日志查询和导出工具。
6. 后续评估是否引入轻量拼音库，并通过 ADR 记录依赖选择。
